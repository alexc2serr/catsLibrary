catsLibrary / Getting Started

Project Overview

catsLibrary (internal name: usj-http-project) is a ground-up implementation of a full HTTP/1.1 client and server stack, written entirely in Node.js utilizing only lower-level transport primitives (net.Socket).

By strictly avoiding high-level abstractions such as Express, Axios, or even the native Node.js http module, this project serves as an extensive study in network protocols, buffer management, socket lifetimes, and RFC compliance.

The Application Domain

On top of this custom HTTP stack, the project implements a fully functional RESTful API for a Cat Shelter application. It supports complex features like relational CRUD operations, file uploads (binary and Base64), and robust session-based authentication.

Core Engineering Objectives

  • Protocol Mastery: Implement HTTP/1.1 message parsing, handling chunked transfer encoding, and Content-Length framing from raw bytes.
  • Resource Management: Properly manage persistent socket connections (keep-alive) and idle timeouts to prevent resource leaks.
  • Security: Implement robust password hashing (PBKDF2), timing-safe comparisons, and secure session cookies.
  • Tooling: Provide both a CLI and a beautiful browser-based GUI to interact with the raw server.

Technology Stack

The codebase is intentionally austere to demonstrate fundamental principles:

  • Runtime: Node.js 18+
  • Language: Vanilla JavaScript (CommonJS modules)
  • Transport Layer: net and tls native modules.
  • Cryptography: crypto native module for hashing and tokens.
  • Testing: Native node:test and node:assert.
  • Dependencies: Zero runtime dependencies for the core protocol. Express is only included as a secondary target to prove protocol interoperability.

catsLibrary / Core HTTP Protocol / HTTP Parser

HTTP Parser Deep Dive

The src/shared/httpParser.js file is the crown jewel of the networking logic. Because we don't rely on Node's HTTP module, we must manually execute string manipulation and buffer reading to understand HTTP/1.1 semantics.

1. Raw String Parsing

The parser uses the strict `\r\n\r\n` (Double CRLF) delimiter to separate the HTTP header section from the HTTP body payload.

const CRLF_DOUBLE = '\r\n\r\n';

function parseRequest(rawRequest) {
  const headerBodySplit = rawRequest.indexOf(CRLF_DOUBLE);
  if (headerBodySplit === -1) {
    throw new Error('Malformed HTTP request: missing header/body separator');
  }

  const headerSection = rawRequest.substring(0, headerBodySplit);
  let body = rawRequest.substring(headerBodySplit + 4);
  
  // ... continued parsing logic ...
}

2. Multi-Value Headers

RFC 9112 specifies that multiple headers with the same name (like Set-Cookie) must be preserved. Our parseHeaders function correctly dynamically upgrades singular values into arrays.

if (headers[name] !== undefined) {
  // Upgrade to array to preserve multiple identical headers
  headers[name] = [].concat(headers[name], value);
} else {
  headers[name] = value;
}

3. Chunked Transfer-Encoding

For payloads of unknown length or extremely large datasets, HTTP uses Transfer-Encoding: chunked. This splits the body into multiple pieces, each prefixed by its hexadecimal size.

Decoding Algorithm

Our decodeChunked algorithm reads the first string until CRLF, parses it via parseInt(line, 16) to determine the chunk size, reads exactly that many bytes, skips the trailing CRLF, and repeats until a chunk size of 0 is encountered.

function decodeChunked(chunkedBody) {
  let result = '';
  let remaining = chunkedBody;

  while (remaining.length > 0) {
    const crlfIdx = remaining.indexOf('\r\n');
    const sizeLine = remaining.substring(0, crlfIdx).split(';')[0];
    const chunkSize = parseInt(sizeLine, 16);
    
    if (isNaN(chunkSize) || chunkSize === 0) break; // last-chunk

    const chunkStart = crlfIdx + 2;
    result += remaining.substring(chunkStart, chunkStart + chunkSize);
    remaining = remaining.substring(chunkStart + chunkSize + 2);
  }
  return result;
}
catsLibrary / Core HTTP Protocol / TCP Server

Raw TCP Socket Server

Implemented in src/server/httpServer.js. This server manages raw TCP sockets using net.createServer. It is responsible for accumulating bytes over the network until a complete HTTP request is formed.

1. Socket Data Buffering

Because TCP is a stream protocol, a single HTTP request might arrive in multiple data events. The server maintains a buffer string for each socket.

server.on('connection', (socket) => {
  let buffer = Buffer.alloc(0);

  socket.on('data', (chunk) => {
    buffer = Buffer.concat([buffer, chunk]);
    
    // Check if headers are fully received
    const splitIdx = buffer.indexOf('\r\n\r\n');
    if (splitIdx === -1) return; // Keep waiting for more data
    
    // Extract Content-Length to determine if body is fully received
    const expectedLength = parseContentLength(buffer);
    const bodyLength = buffer.length - (splitIdx + 4);
    
    if (bodyLength >= expectedLength) {
      // Request is fully received! We can now process it.
      handleRequest(socket, buffer.toString('utf8'));
      buffer = Buffer.alloc(0); // Reset for next keep-alive request
    }
  });
});

2. Persistent Connections (Keep-Alive)

To improve latency, HTTP/1.1 uses persistent connections by default. The server checks the parsed Connection header.

  • If it equals close, we call socket.end() after writing the response.
  • Otherwise, we inject Connection: keep-alive into the response and keep the socket open.

Idle Timeout Defense

To prevent malicious clients from holding sockets open indefinitely (Slowloris attack), the server attaches a 30-second timeout to the socket using socket.setTimeout(30000). If no data arrives, the socket is forcibly destroyed.

3. Handling Binary Data (Images)

When clients upload cat photos (using binary bodies rather than base64), standard UTF-8 parsing corrupts the bytes. The server detects Content-Type: image/* and avoids calling .toString('utf8') on the body, preserving the raw binary Buffer for file storage.

TLS Security & Express Bridge

TLS Secure Server (HTTPS)

While the standard server runs on unencrypted TCP, we implemented a secure variant in httpServerTLS.js. This utilizes Node.js's tls module.

Certificate Generation:
We use an automated script scripts/generateCerts.js which spawns an openssl process to generate a self-signed RSA-2048 keypair.

const tls = require('tls');
const options = {
  key: fs.readFileSync('certs/server.key'),
  cert: fs.readFileSync('certs/server.cert')
};

const secureServer = tls.createServer(options, (socket) => {
  // The exact same buffering and HTTP parsing logic from httpServer.js is injected here!
  attachHttpHandlers(socket); 
});
secureServer.listen(3443);

Express Interoperability Bridge

To definitively prove that our custom httpClient.js is 100% RFC compliant, we built httpServerExpress.js. This server exposes the identical REST API using the popular Express.js framework.

You can start it on port 3001, point the CLI client to it, and all commands (CRUD, Auth, Cookies, Photo Uploads) will work flawlessly, proving our custom client speaks perfect standard HTTP.

catsLibrary / Backend Services / Authentication System

Authentication & Session System

Security is paramount. The authRoutes.js module manages user registration, secure password hashing, and session persistence.

1. Cryptography & Hashing

Passwords are never stored in plaintext. We utilize PBKDF2 (Password-Based Key Derivation Function 2) with HMAC-SHA256.

  • Salt: 16 bytes of random data per user.
  • Iterations: 100,000 iterations to deliberately slow down brute-force attacks.
  • Key Length: 64 bytes.
const crypto = require('crypto');

function hashPassword(password, salt) {
  return crypto.pbkdf2Sync(password, salt, 100000, 64, 'sha256').toString('hex');
}

// During Login: Preventing Timing Attacks
const inputHash = hashPassword(reqPassword, user.salt);
const isValid = crypto.timingSafeEqual(
  Buffer.from(inputHash), 
  Buffer.from(user.hash)
);

2. Session Tokens & Cookies

Upon successful login, a cryptographically secure 32-byte token is generated. This token is mapped to the user ID in server memory.

The server delivers this token back to the client via the Set-Cookie header with strict security directives:

Set-Cookie: sessionToken=abc123def456...; HttpOnly; Path=/; Max-Age=3600; SameSite=Strict

Because it is HttpOnly, malicious JavaScript running in the browser (XSS) cannot steal the token.

Conditional GET & ETag Caching

To save network bandwidth, the server calculates an MD5 hash of the response payload and attaches it as the ETag header.

const eTag = crypto.createHash('md5').update(jsonResponse).digest('hex');
headers['ETag'] = `"${eTag}"`;

if (request.headers['if-none-match'] === `"${eTag}"`) {
  // The client's cached version is identical to the current data!
  return serializeResponse({ statusCode: 304, statusText: 'Not Modified' });
}

If the client sends If-None-Match: "hash" and it matches, the server aborts sending the body and responds with 304 Not Modified.

catsLibrary / Backend Services / Owners API

Owners API (Advanced Relational CRUD)

To demonstrate advanced database mapping capabilities, ownerRoutes.js implements a secondary resource with a bidirectional relationship to Cats.

Bidirectional Foreign Keys

Cats contain an ownerId property. When an Owner is requested via GET /api/owners/:id, the server dynamically queries the Cats table and embeds the full Cat objects into the Owner response payload.

Relational Endpoints

Assigning and unassigning cats is handled via specific nested sub-routes, treating the relationship itself as an action.

MethodPathDescription
POST/api/owners/:oid/cats/:cidAssigns Cat :cid to Owner :oid. Updates the ownerId foreign key.
DELETE/api/owners/:oid/cats/:cidRemoves the assignment (sets ownerId = null).
DELETE/api/owners/:idDeletes the Owner. Crucially, the server performs a cascading update to unlink all associated cats first to prevent orphaned foreign keys.
catsLibrary / Backend Services / Middleware

Middleware & Logging

The middleware.js file acts as an interceptor pipeline. Before a request ever reaches a route handler (like Cats or Owners), it must pass through the middleware chain.

1. Authentication Interceptor

The authenticate() function protects all routes starting with /api/.

It checks three distinct sources in order of precedence:

  1. The X-API-Key custom header (matched against a hardcoded admin key).
  2. The Authorization: Bearer <token> header.
  3. The parsed Cookie header looking for sessionToken.

If none of these are valid, the middleware short-circuits the pipeline and returns a 401 Unauthorized response directly.

2. Apache Combined Logger

The logRequest() function records every interaction. It extracts the timestamp, client IP, method, requested path, HTTP version, response status code, content length, and the User-Agent.

function logRequest(req, res, clientIp) {
  const logLine = `[${new Date().toISOString()}] ${clientIp} "${req.method} ${req.path} HTTP/${req.httpVersion}" ${res.statusCode} ${res.body.length} "-" "${req.headers['user-agent'] || '-'}"\n`;
  
  // Asynchronously append to file
  fs.appendFile('logs/access.log', logLine, (err) => { ... });
  
  // Output to stdout with ANSI colors
  console.log(`\x1b[36m${req.method}\x1b[0m \x1b[32m${res.statusCode}\x1b[0m ${req.path}`);
}
catsLibrary / Client Tools / Terminal CLI

Terminal CLI Client

Because standard command-line tools like curl or Postman hide the underlying HTTP mechanics, we built our own native Node.js client logic.

HTTP Client Library (httpClient.js)

This library connects to servers using net.createConnection(). It mimics the interface of modern libraries (like fetch or Axios) but executes the manual serialization logic built in our httpParser.js.

The CookieJar Implementation

Unlike browsers, raw TCP sockets do not remember cookies between requests. httpClient.js implements a CookieJar class. When it receives a Set-Cookie header, it parses the attributes (Path, Expires, Max-Age), stores it in memory, and automatically injects it as a Cookie: header into all subsequent requests to the same domain.

Interactive CLI (index.js)

The user-facing terminal interface utilizes Node's readline module to create an interactive menu.

catsLibrary / Engineering / RFC 9112

RFC 9112 Compliance Specifications

The primary grading criteria for this project is strict adherence to HTTP/1.1 RFC 9112 specifications. Our implementation ensures broad compatibility.

RFC 9112 RequirementSectionOur Technical Implementation
CRLF Framing§2.2httpParser.js uses strict \r\n constants for all line endings and \r\n\r\n to terminate headers.
Request-Line Format§3.1Method, target, and HTTP-version are split precisely using space delimiters (indexOf(' ')).
Content-Length Calculation§6.2Server dynamically calculates payload size using Buffer.byteLength(body) to account for multi-byte Unicode characters instead of naive .length.
Transfer-Encoding§7.1Full bidirectional Chunked encoding/decoding is implemented. The parser successfully strips chunk extensions and trailers.

catsLibrary / Engineering / Design Decisions

System Design Decisions

1. String vs Buffer Memory Management

The Challenge: Managing raw memory efficiently when receiving arbitrary network payloads.

The Decision: We opted to use Node's Buffer.concat() exclusively while accumulating the payload. We only call .toString('utf8') when we are 100% certain the request is fully formed and ready to be routed. This prevents memory leaks and corruption of binary file uploads.

2. Dependency Injection for Cross-Module Routing

The Challenge: middleware.js needs to validate session tokens, which are generated and managed inside authRoutes.js. Directly importing authRoutes.js into middleware.js created a circular dependency.

The Decision: We implemented a Dependency Injection pattern. The main httpServer.js bootstrapper pulls the validateToken function from Auth and injects it into Middleware via setTokenValidator(fn) during startup.

3. ETag Header Injection

The Challenge: serializeResponse() formats the final HTTP string. However, calculating the ETag requires the final JSON body. We needed a way to inject headers late in the lifecycle without rebuilding the entire object.

The Decision: We created an injectHeaders(rawString, headers) utility that uses regex to find the \r\n\r\n boundary and seamlessly splices new headers into the pre-compiled string. This significantly improved performance by avoiding double-serialization.

4. Avoiding HTTP/2 Upgrade Mechanics

The Challenge: Modern clients often request Upgrade: h2c.

The Decision: Due to the complexity of HPACK and binary framing, our server intentionally ignores Upgrade headers and forces fallback to standard HTTP/1.1 per the project scope.
